Abstract

RetroQuest is a retro-inspired domain-specific language designed for text-based adventure games and branching interactive fiction. The project demonstrates all major compiler phases including lexical analysis, syntax analysis, semantic analysis, intermediate representation generation, optimization, and execution through a command-line interpreter. Source files written in the .rq language are translated into a structured JSON story graph, which is then executed by a lightweight runtime engine. The project emphasizes simple language design, clear compiler architecture, and creative application of compiler construction concepts in an interactive storytelling domain.

1. Introduction

Compiler Construction is often taught using arithmetic expressions or small toy languages. In this project, we chose a more creative direction by designing a mini language for retro-style interactive fiction. RetroQuest allows users to define scenes, choices, variable updates, and conditional branches in a compact syntax. The main purpose of the project is to map all compiler phases to a working implementation that is easy to understand, demonstrate, and extend.

2. Language Design and Motivation

The project theme is “Choose Your Own Adventure DSL.” We selected this theme because it naturally supports branching control flow, labels, semantic validation, and custom execution logic. The language is designed to be beginner-friendly and intentionally small so that compiler phases remain clear.

  • File extension: .rq
  • Target output: JSON story graph
  • Execution model: command-line interpreter

3. Lexical Design

The lexer scans the source program and recognizes keywords, identifiers, literals, operators, punctuation, comments, and whitespace. Each token is emitted with line and column information for accurate error reporting.

Token Class Examples
Keywords game, var, scene, show, choice, goto, set, if, then, end, restart
Identifiers start, has_key, freedom
Literals strings, integers, booleans
Operators / Symbols =, ==, !=, ->, +, :

4. Syntax Design and Grammar

RetroQuest follows a scene-oriented grammar. A program may optionally begin with a game title, followed by global variable declarations and one or more scene declarations.

<program>         ::= [<game_decl>] {<var_decl>} {<scene_decl>}
<game_decl>       ::= "game" <string>
<var_decl>        ::= "var" <identifier> "=" <expr>
<scene_decl>      ::= "scene" <identifier> ":" {<statement>}

<statement>       ::= <show_stmt>
                    | <choice_stmt>
                    | <goto_stmt>
                    | <set_stmt>
                    | <if_goto_stmt>
                    | <end_stmt>
                    | <restart_stmt>

<expr>            ::= <primary_expr> { "+" <primary_expr> }

5. Semantic Rules and Symbol Table

  • All variables must be declared before use.
  • All target scenes must exist.
  • Duplicate variable names are rejected.
  • Duplicate scene labels are rejected.
  • The language requires a start scene as the entry point.
  • Addition is allowed only for integer operands.

6. Intermediate Representation

The compiler generates a JSON-based story graph. Each scene becomes a node and transitions are represented by choices or goto edges. This IR was selected because it closely matches the semantics of interactive fiction and keeps the runtime simple.

{
  "title": "Treasure Hunt",
  "variables": {
    "has_key": false
  },
  "scenes": {
    "start": [
      { "type": "ShowStatement", "value": "You wake up in a dark room." },
      { "type": "ChoiceStatement", "text": "Open the wooden chest", "target": "chest" }
    ]
  },
  "entry_scene": "start"
}

7. Optimization Techniques

The project implements two basic optimizations:

  • Constant folding / propagation
  • Unreachable code and unreachable scene elimination

Before Optimization

if open_gate then goto ending
show "This line is unreachable."
end

After Optimization

goto ending

8. Code Generation and Interpretation

Instead of generating machine code, RetroQuest compiles source programs into an optimized story graph and executes them through a custom command-line interpreter. This approach is valid for the project brief because the target output may be interpreted rather than native machine code.

10. Challenges and Limitations

One challenge was keeping the grammar simple while still supporting useful story mechanics. Another challenge was avoiding unsafe optimization of mutable variables. The current version supports global scope only and limited arithmetic, so it is intentionally focused rather than general-purpose.

11. Conclusion

RetroQuest successfully demonstrates how a custom language can be designed and compiled through all major phases of compiler construction. The project combines creativity with practical compiler design and provides a working implementation that is easy to demonstrate in a viva setting.

12. Team Reflection

During the development of RetroQuest, our team learned how the theoretical phases of compiler construction map to a practical software system. At the start, the biggest challenge was defining a grammar that remained simple enough for recursive descent parsing while still supporting useful features such as scenes, choices, conditions, and variable updates. We solved this by keeping the language intentionally focused and by designing a clean scene-based structure.

Another challenge was semantic analysis. It was not enough to parse valid syntax; we also needed to detect undefined scenes, duplicate declarations, and invalid variable usage. This phase helped us understand the importance of symbol tables and validation beyond grammar rules.

We also learned that intermediate representation design is a major compiler decision. Instead of generating machine code, we chose a JSON story graph because it matched our language domain and made execution easier through a command-line interpreter. This gave us a clear separation between compilation and runtime behavior.

If we had more time, we would extend RetroQuest with richer arithmetic, nested conditions, inventory lists, and more advanced optimization passes.
