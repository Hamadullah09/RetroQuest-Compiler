State Input Class Next State Meaning / Accepted Token
S0 letter / underscore S1 Start identifier / keyword
S1 letter / digit / underscore S1 Continue identifier
S1 delimiter / whitespace Accept IDENTIFIER or KEYWORD
S0 whitespace S0 Ignore spaces, tabs, newlines

S3 any valid character except " S3 Continue string

S3 " Accept STRING
S3 newline / EOF Error Unterminated string

Handwritten tip: draw states S0 to S8. This is enough for the lexer deliverable because the brief asks for token recognition DFA/NFA, not a DFA for parsing or semantic analysis.

Accepting States Summary

Accepted Token Recognized From Notes
IDENTIFIER / KEYWORD S1 on delimiter Keyword check is performed after lexeme collection
NUMBER S2 on delimiter Only integer literals supported
STRING S3 on closing quote Escape sequences route through S8
ASSIGN S4 on non-= lookahead Single =
EQ S4 followed by = Double equals
NEQ S5 followed by = Not-equal operator
ARROW S6 followed by > Choice branch operator
PLUS S0 Addition operator
COLON S0 Scene label separator
EOF S0 at end of stream End marker token

How To Draw The Lexer DFA By Hand

  1. Draw S0 as the lexer start state.
  2. From S0, draw outgoing arrows for identifier, number, string, operator, comment, and whitespace classes.
  3. Use S1 for identifier/keyword recognition.
  4. Use S2 for integer literal recognition.
  5. Use S3 and S8 for string literal and escape handling.
  6. Use S4, S5, and S6 for multi-character operators.
  7. Use S7 for comment skipping.
  8. Show clear accepting outputs: IDENTIFIER/KEYWORD, NUMBER, STRING, ASSIGN, EQ, NEQ, ARROW, PLUS, COLON, EOF.

5. Optimization Example

This is useful for the “before/after optimization” part of the brief.

Source Program

game "Fixed Ending"

var open_gate = true

scene start:
if open_gate then goto ending
show "This line is unreachable."
end

scene ending:
show "The gate is already open."
end

IR Before Optimization

{
  "start": [
    { "type": "IfGotoStatement", "condition": { "variable": "open_gate" }, "target": "ending" },
    { "type": "ShowStatement", "value": "This line is unreachable." },
    { "type": "EndStatement" }
  ],
  "ending": [
    { "type": "ShowStatement", "value": "The gate is already open." },
    { "type": "EndStatement" }
  ]
}

IR After Optimization

{
  "start": [
    { "type": "GotoStatement", "target": "ending" }
  ],
  "ending": [
    { "type": "ShowStatement", "value": "The gate is already open." },
    { "type": "EndStatement" }
  ]
}

Optimization Notes

  • Constant folding / propagation: open_gate is known to be true, so the condition is resolved at compile time.
  • Unreachable code elimination: the show statement and following end after the unconditional branch are removed.

Abstract Draft

RetroQuest is a retro-inspired domain-specific language designed for text-based adventure games and branching interactive fiction. The project demonstrates all major compiler phases including lexical analysis, syntax analysis, semantic analysis, intermediate representation generation, optimization, and execution through a command-line interpreter. Source files written in the .rq language are translated into a structured JSON story graph, which is then executed by a lightweight runtime engine. The project emphasizes simple language design, clear compiler architecture, and creative application of compiler construction concepts in an interactive storytelling domain.

7. Team Reflection Draft

Use and edit this in your report. You can shorten or personalize it.

During the development of RetroQuest, our team learned how the theoretical phases of compiler construction map to a practical software system. At the start, the biggest challenge was defining a grammar that remained simple enough for recursive descent parsing while still supporting useful features such as scenes, choices, conditions, and variable updates. We solved this by keeping the language intentionally focused and by designing a clean scene-based structure.

Another challenge was semantic analysis. It was not enough to parse valid syntax; we also needed to detect undefined scenes, duplicate declarations, and invalid variable usage. This phase helped us understand the importance of symbol tables and validation beyond grammar rules.

We also learned that intermediate representation design is a major compiler decision. Instead of generating machine code, we chose a JSON story graph because it matched our language domain and made execution easier through a command-line interpreter. This gave us a clear separation between compilation and runtime behavior.

The optimization phase was also valuable. By implementing constant folding and unreachable code elimination, we observed how even simple optimizations can improve the generated representation and make compiler output cleaner.

If we had more time, we would extend RetroQuest with richer arithmetic, nested conditions, inventory lists, and more advanced optimization passes. Overall, this project gave us hands-on understanding of how a compiler is designed, implemented, debugged, and demonstrated.

Example 3: Dungeon Escape

game "Dungeon Escape"
var has_key = false

scene start:
show "You wake up in an old dungeon."
choice "Open the chest" -> chest
choice "Try the iron door" -> door

scene chest:
show "Inside the chest you find a rusty key."
set has_key = true
goto start

scene door:
if has_key then goto freedom
show "The door is locked."
end

scene freedom:
show "You unlock the door and escape. You win!"
end
